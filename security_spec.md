# Security Specification - Work Tracker

## Data Invariants
1. **Ownership**: A work log entry MUST belong to the user who created it (`uid` field matches `auth.uid`).
2. **Integrity**: Timestamps (`createdAt`) MUST be generated by the server, not the client.
3. **Verified Access**: Users MUST have a verified email to perform write operations (to prevent bot spam).
4. **Immutability**: Once a log is created, it cannot be modified (following the "Update-Gap" prevention strategy, and as requested by the user for simplicity, deletion is the only way to "fix" errors).

## "Dirty Dozen" Payloads (Red Team Tests)
1. **Identity Spoofing**: Attempt to create a log with `uid` belonging to another user.
2. **Timestamp Fraud**: Attempt to set `createdAt` to a date in the past/future.
3. **Shadow Field Injection**: Attempt to include a field `isAdmin: true` in the `logs` collection.
4. **Large Payload**: Attempt to inject a 1MB string into the `start` or `end` time fields.
5. **ID Poisoning**: Attempt to use an extremely long or invalid character string as the Document ID.
6. **Anonymous Write**: Attempt to create a log without being signed in.
7. **Unverified Email**: Attempt to write with a brand new account that hasn't verified its email.
8. **Malicious Query**: Attempt to list ALL logs from ALL users by omitting the `where` clause in the query.
9. **Total field bypass**: Attempt to send a negative `total` hour count.
10. **Key Mismatch**: Attempt to send a log missing the `date` field.
11. **Type Mismatch**: Attempt to send `total` as a string instead of a number.
12. **Cross-User Deletion**: Attempt to delete another user's log by guessing its ID.
